# Low Level Design — Prompt Assembly and Model Routing Layer

## 1. Purpose

This layer is responsible for converting a raw chat request plus resolved assistant policy into a **model-ready prompt** and a **model execution decision**.

It has two closely related responsibilities:

* **Prompt Assembly**

  * build final system, developer, and user prompt structure
  * resolve variables from request, assistant config, tenant context, and runtime flags
  * inject RAG / memory / tool / response-shaping instructions
  * apply prompt templates consistently

* **Model Routing**

  * classify the request
  * determine complexity and execution shape
  * select provider + model + chat options
  * apply routing and fallback policy

This maps cleanly to Spring AI because prompt creation is built around `Prompt`, `Message`, and `PromptTemplate`, while execution options can be attached via `ChatOptions` / model-specific options. Spring AI’s `ChatClient` is the final execution API, but prompt/template construction and model choice are application responsibilities above that API. ([Home][1])

---

## 2. Scope of the Layer

This layer covers:

* prompt template resolution
* placeholder variable substitution
* system prompt construction
* developer prompt construction
* user prompt normalization
* response formatting instruction injection
* tool-usage instruction injection
* grounded-answer instruction injection
* routing classification
* model selection
* chat option selection
* fallback model planning
* prompt/model validation

It does **not** cover:

* config persistence
* JPA entities
* execution orchestration
* direct `ChatClient` invocation
* memory repository implementation
* vector retrieval execution
* tool execution implementation

## 4. Design Principles

## 4.1 Prompt assembly must be deterministic

Given:

* same resolved config
* same request
* same runtime override

the layer should produce the same final prompt payload.

That is critical for auditability and reproducibility.

---

## 4.2 Routing must be policy-driven, not ad hoc

Avoid:

```java id="zbzj13"
if (message.length() > 3000) return "gpt-4";
else return "gpt-4-mini";
```

Use:

* explicit route definitions
* ordered rule evaluation
* clear route result metadata

---

## 4.3 Prompt structure and model routing must be separate concerns

Even though they work together, keep them separate:

* prompt package decides **what to say to the model**
* modelrouting package decides **which model and options to use**

The orchestration layer will call both in sequence.

---

## 4.4 Build Prompt objects, not only raw strings

Spring AI’s prompt API is centered on `Prompt`, `Message`, and `PromptTemplate`, and templates support variable expansion plus optional model options during creation. This favors building structured prompt objects or structured prompt parts, not treating everything as one raw concatenated string. ([Home][1])

---

## 4.5 Prompt augmentation should be compatible with Advisors

Spring AI notes that prompt-augmenting advisors use self-contained templates and the Advisors API is specifically designed to intercept and enhance interactions. Your own prompt layer should therefore avoid brittle shared-template assumptions and should produce prompt structures that coexist cleanly with memory/RAG/tool advisors. ([Home][2])

---

# PART A — Prompt Assembly Layer

## 6. Purpose of Prompt Assembly

The prompt layer builds the final prompt content from:

* assistant prompt templates
* assistant guardrails
* response policy
* runtime context
* user request
* selected runtime flags
* optionally, instructions related to tools, memory, and grounded answering

It should produce a **normalized prompt result** that the orchestration layer can pass into Spring AI `ChatClient`.

## 10. Prompt Composition Strategy

## 10.1 Final prompt structure

Recommended final structure:

### System Prompt

1. rendered base system prompt template
2. guardrail instructions
3. safety instructions
4. grounded-answer instructions
5. output formatting instructions

### Developer Prompt

1. rendered developer prompt template
2. tool usage rules
3. citation policy
4. domain-specific execution notes

### User Prompt

1. normalized user message
2. optionally appended light context summary

---

## 10.2 Prompt merge order

```text id="zfyz5e"
Base System Template
   + Guardrails
   + Safety Instructions
   + Grounded Answer Instructions
   + Response Format Instructions
   = Final System Prompt

Base Developer Template
   + Tool Usage Instructions
   + Citation Instructions
   + Runtime Behavioral Notes
   = Final Developer Prompt
```

---

## 10.3 Grounded-answer prompt policy

If `groundedAnswerRequired = true`, inject instructions such as:

* answer only from supplied knowledge context
* if context is insufficient, say so
* do not invent policies or facts
* cite where applicable

This is application policy, while Spring AI RAG integration later supplies the actual retrieved context through advisors. Spring AI’s retrieval augmentation is advisor-driven, so the prompt layer should only inject the behavioral instruction, not the retrieval payload itself. ([Home][2])

---

## 10.4 Tool instruction policy

If tool calling is enabled:

* instruct the model to use tools when needed
* state that only exposed tools are available
* instruct not to fabricate tool results
* prefer tool calls for certain task classes if configured

Spring AI’s tool-calling support includes both framework-controlled execution and advisor-controlled execution via `ToolCallAdvisor`; whichever route you use, the model still benefits from explicit behavior instructions in the prompt. ([Home][3])

# PART B — Model Routing Layer

## 14. Purpose of Model Routing

The model-routing layer selects the best model and execution options for a request.

It uses:

* request characteristics
* assistant routing policy
* runtime override hints
* feature requirements such as tool use, streaming, grounded mode, or longer context

This is necessary because Spring AI can work with multiple chat models and the actual model choice is up to the application. The framework offers the unified chat abstractions, while your platform decides which model implementation to use for a given request. ([Home][4])

## 17. Main Routing Classes

## 17.1 `DefaultModelRoutingService`

```java id="2cdh5v"
@Service
public class DefaultModelRoutingService implements ModelRoutingService {

    private final RequestClassifier requestClassifier;
    private final RoutingPolicyEvaluator routingPolicyEvaluator;
    private final ModelSelectionValidator modelSelectionValidator;

    @Override
    public ModelSelectionResult select(RoutingInput input) {
        RequestClassification classification = requestClassifier.classify(input);
        ModelSelectionResult result = routingPolicyEvaluator.evaluate(input, classification);
        modelSelectionValidator.validate(result, input);
        return result;
    }
}
```

## 18. Routing Strategy

## 18.1 Ordered route evaluation

Use route priority from config.

Pseudo-flow:

```text id="9r6dqu"
1. get active routes ordered by priority
2. for each route:
      if route matches classification and request constraints:
          return selected route
3. else return default model
4. apply fallback policy
```

---

## 18.2 Matching criteria examples

A route may match on:

* request complexity
* prompt size range
* `ragEnabledOnly`
* `toolsRequiredOnly`
* `structuredOutputOnly`
* channel
* tenant policy
* explicit model hint allowlist

---

## 18.3 Example routes

```text id="d9ha2y"
Route 1: SIMPLE_QA
- prompt length < 500
- no RAG
- no tools
- use small/cheap model

Route 2: GROUNDED_QA
- RAG required
- citations enabled
- use stronger reasoning model

Route 3: TOOL_HEAVY
- tool usage expected
- use model with reliable tool calling

Route 4: LONG_CONTEXT
- input length > threshold
- use model with larger context window
```

Spring AI’s chat abstractions are model-neutral, but model capabilities still vary by provider and endpoint. For example, the Bedrock Converse API page highlights support for tool calling, system messages, streaming, and multimodal inputs across supported providers. That is exactly why your routing layer should choose models by capability, not only by cost. ([Home][5])

## 22. Prompt and Routing Integration Sequence

```text id="sj9q4f"
1. Orchestration receives ChatRequest
2. Config layer returns ResolvedAssistantConfig
3. ModelRoutingService.select(...)
4. returns ModelSelectionResult
5. PromptAssemblyService.assemble(...)
   using:
   - request
   - resolved config
   - model selection result
6. Orchestration builds execution plan
7. ChatOptionsFactory builds ChatOptions
8. ChatClient executes request
```

### Why model routing before prompt assembly?

Because prompt assembly may depend on selected model characteristics:

* max context
* stricter formatting
* streaming enabled
* tool-calling support assumptions

## 23. Prompt / Routing Interaction with Advisors

Spring AI’s Advisors API is designed for reusable augmentation and interception, and built-in advisors such as memory and RAG are expected to participate in prompt enrichment and iterative flows. The tool-calling advisor also has its own memory/history behavior, which can be disabled when you want chat memory to remain the single source of history management. ([Home][2])

### Design implication

This layer should:

* define instructions and options
* not duplicate memory/RAG payload injection done by advisors
* remain compatible with advisor-based augmentation

### Good separation

* prompt layer: “answer only from supplied context”
* RAG advisor: supplies the actual retrieved context
* memory advisor: injects conversation memory
* tool advisor/framework: runs tool loop
