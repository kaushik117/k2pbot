# Low Level Design — API Request/Response Schemas

## 1. Purpose

The API Schema Layer defines the external contracts exposed by the chatbot platform.

Its purpose is to standardize:

* chat request payloads
* chat response payloads
* streaming response payloads
* session and history APIs
* config/admin read APIs
* error response structures
* validation rules
* field-level semantics

This layer is the boundary between:

* client channels such as web/mobile/API consumers
* the orchestration runtime

It should be stable, versioned, and independent from internal entity structure.

---

## 2. Scope of the API Schema Layer

This layer covers:

* REST request DTOs
* REST response DTOs
* streaming event payloads
* validation annotations
* response metadata contracts
* error response contracts
* pagination request/response DTOs
* admin/config query DTOs if exposed

It does **not** cover:

* controller business logic
* JPA entities
* runtime execution objects
* internal config models
* persistence mappings

## 4. API Design Principles

## 4.1 Separate API DTOs from internal runtime objects

Do not expose:

* `ExecutionContext`
* `ResolvedAssistantConfig`
* JPA entities
* internal audit entities

Use dedicated DTOs only.

---

## 4.2 Stable versioned contracts

All APIs should be versioned.

Recommended base path:

```text id="o0zc7r"
/api/v1
```

Example:

* `/api/v1/chat`
* `/api/v1/chat/stream`
* `/api/v1/sessions/{sessionId}`
* `/api/v1/sessions/{sessionId}/messages`

---

## 4.3 Minimal required request fields

Do not force clients to send too many internal details.

Required fields should be minimal:

* assistant code
* tenant ID
* session ID
* user message

Optional:

* user ID
* locale
* channel
* context
* runtime overrides

---

## 4.4 Explicit metadata in responses

Every response should clearly state:

* session ID
* message ID
* selected model
* timestamp
* citations if applicable
* tool execution summary if applicable

---

## 4.5 Streaming and non-streaming contracts should be separate

Do not overload one schema for both sync and streaming.

Recommended:

* sync returns one final response object
* streaming returns chunk/event DTOs

---

## 4.6 Consistent error contract

All API failures should return one normalized error schema.

## 6. Request DTOs

## 6.1 ChatRequestDto

```java id="fwkbr4"
public class ChatRequestDto {

    @NotBlank
    @Size(max = 100)
    private String assistantCode;

    @NotBlank
    @Size(max = 100)
    private String tenantId;

    @NotBlank
    @Size(max = 100)
    private String sessionId;

    @NotBlank
    @Size(max = 100)
    private String userId;

    @NotBlank
    @Size(max = 20000)
    private String message;

    @Size(max = 20)
    private String locale;

    @Size(max = 50)
    private String channel;

    private Map<String, Object> context;

    @Valid
    private RuntimeOverrideDto runtimeOverride;
}
```

### Field notes

* `assistantCode` → identifies runtime specialization
* `tenantId` → tenant context
* `sessionId` → conversation continuity
* `userId` → audit and personalization anchor
* `message` → user prompt
* `context` → optional business/channel metadata
* `runtimeOverride` → restricted per-request override block

---

## 6.2 RuntimeOverrideDto

```java id="6m8gu8"
public class RuntimeOverrideDto {

    @Size(max = 100)

    private MemoryStoreType memoryStoreType;

    @Size(max = 100)

    private Boolean streamingEnabled;
}
```

### Rule

This DTO is only a **request hint**.
The server must validate and restrict all values.

## 12. Validation Rules

## 12.1 Standard Bean Validation

Use:

* `@NotBlank`
* `@Size`
* `@Min`
* `@Max`
* `@Valid`

---

## 12.2 Custom validators

Add custom validators where needed.

Examples:

* `@AllowedChannel`
* `@AllowedLocale`
* `@ValidRuntimeOverride`
* `@ValidToolNames`

### Example custom validator target

`RuntimeOverrideDto`

* reject tool list if empty but provided
* reject unsupported memory mode
* reject invalid model hint format

---

## 12.3 Message validation policy

Recommended:

* trim whitespace
* reject blank-only content
* set max input length
* optionally reject extremely large context maps

## 16. Security-Relevant API Considerations

### Never expose

* raw prompt templates
* internal stack traces
* complete tool payloads with secrets
* internal policy rule structures
* raw config override payloads from DB

### Sanitize

* error messages
* tool failure details
* source snippets if sensitive

### Optional headers

Recommended headers:

* `X-Request-Id`
* `X-Tenant-Id` if not present in body, though for your use case body field is simpler
* authentication headers outside API schema scope

## 19. Suggested Internal-to-External Mapping Boundaries

### Web layer DTOs

* input and output only

### Orchestration layer

* internal request/response objects

### Persistence layer

* entities

### Config layer

* resolved runtime config objects

Do not cross these boundaries.

---

## 20. Contract Evolution Guidelines

### Additive changes preferred

Safe:

* adding new optional response field
* adding new metadata field
* adding new event type

Risky:

* renaming fields
* changing enum meaning
* making optional fields required

### Recommendation

Use explicit API versioning if breaking changes are needed.

---

## 21. Recommended Implementation Order

Build in this order:

1. request DTOs
2. response DTOs
3. error DTOs
4. controller interfaces
5. mappers
6. exception advice
7. custom validators
8. streaming DTOs
