Below is the detailed **Low Level Design for the Persistence / Database Layer** using **Spring Data JPA** in your **single Maven project** structure.

---

# Low Level Design — Persistence / Database Layer

## 1. Purpose

The Persistence Layer is responsible for storing and retrieving all durable application data needed by the chatbot platform, especially:

* assistant configuration
* prompt versions and policies
* tool policies
* model-routing policies
* knowledge base registry metadata
* chat session metadata
* chat message history
* execution audit records
* tool execution audit records
* optional memory persistence metadata

Since you want a database-backed design, **Spring Data JPA** is the right abstraction for the relational parts of this platform. Spring Data JPA provides repository support for Jakarta Persistence with a consistent programming model, and repositories are standard Spring beans by default. ([Home][1])

This layer should remain a **pure persistence layer**. It must not:

* call LLMs
* assemble prompts
* execute tools
* run RAG retrieval
* contain orchestration branching

---

## 2. Scope of the Persistence Layer

This layer covers:

* JPA entities
* repository interfaces
* entity relationships
* transactional boundaries for write operations
* optimistic concurrency/version handling where required
* query methods for config lookup, chat history lookup, and audit lookup
* soft-delete or active-flag patterns where useful
* schema constraints and indexes
* mapping of durable records to domain persistence objects

It does **not** cover:

* API schemas
* runtime config resolution
* Spring AI `ChatClient` assembly
* `ChatMemory` object construction
* vector similarity retrieval itself

---

## 3. Package Structure

```text id="1av5q7"
com.yourcompany.ai.chatbot
├── persistence
│   ├── entity
│   ├── repository
│   ├── projection
│   ├── specification
│   ├── converter
│   ├── support
│   ├── service
│   ├── exception
│   └── config
```

### Package responsibility split

* `persistence.entity` → JPA entities
* `persistence.repository` → Spring Data JPA repositories
* `persistence.projection` → read-only projections for optimized queries
* `persistence.specification` → dynamic criteria/specifications if needed
* `persistence.converter` → attribute converters for JSON/enums/structured values
* `persistence.support` → base entities, constants, common enums
* `persistence.service` → persistence-focused transactional services
* `persistence.exception` → DB/persistence-related exceptions
* `persistence.config` → JPA configuration, auditing config, converters registration

## 4. Design Principles

## 4.1 Separate persistence model from runtime model

Do not use JPA entities as runtime orchestration objects.

Keep:

* `ResolvedAssistantConfig` in config layer
* `ExecutionContext` in orchestration layer
* JPA entities only for persistence

This avoids:

* lazy-loading issues in runtime logic
* accidental entity mutation
* persistence concerns leaking into orchestration

---

## 4.2 Use JPA for relational operational data

Use Spring Data JPA for:

* assistant config
* prompt config
* model policies
* tool policies
* session metadata
* message history
* execution audit

This fits the structured, relational nature of these records. Spring Data JPA is designed to reduce boilerplate for JPA-based persistence and supports repository-style access cleanly. ([Home][1])

---

## 4.3 Keep vector data out of core JPA entities

If you use a vector store such as PGvector later, treat vector embeddings/search as part of the **RAG layer**, not the core relational persistence model. Spring AI exposes a separate portable Vector Store API for vector retrieval concerns. ([Home][2])

So:

* knowledge base metadata can live in JPA tables
* actual vector documents/search belong to the vector-store integration

---

## 4.4 Transaction boundaries belong in services, not controllers

Spring Data JPA supports transactional repository methods, but declared query methods do not automatically get transaction configuration unless you add it. In practice, transaction boundaries should be managed at service/facade level for consistency. ([Home][3])

Recommended:

* read-only service methods → `@Transactional(readOnly = true)`
* write methods → `@Transactional`

---

## 4.5 Audit-friendly schema

Every important write should be attributable to:

* tenant
* assistant
* session
* request
* config version
* selected model

This matters because Spring AI observability does not replace your application-level business audit. Spring AI can emit observations and optionally log prompt/completion content, but prompt logging is off by default and sensitive by nature. ([Home][4])

---

## 5. Data Domains in Persistence

Split persistence into four logical areas:

### A. Configuration domain

Stores durable assistant/platform configuration.

Tables:

* `ai_assistant`
* `ai_assistant_prompt`
* `ai_assistant_model_route`
* `ai_assistant_memory_policy`
* `ai_assistant_rag_policy`
* `ai_assistant_tool_policy`
* `ai_assistant_safety_policy`
* `ai_assistant_response_policy`
* `ai_knowledge_base`
* `ai_tenant_assistant_override`

### B. Conversation domain

Stores user conversation state/history.

Tables:

* `ai_chat_session`
* `ai_chat_message`

### C. Execution audit domain

Stores execution metadata and debugging/audit data.

Tables:

* `ai_chat_execution`
* `ai_tool_execution_audit`
* `ai_rag_retrieval_audit`

### D. Optional memory persistence domain

Only if you persist memory state separately from raw history.

Tables:

* `ai_chat_memory_snapshot`
* or repository-specific memory store tables

Spring AI distinguishes **chat memory** from **chat history**, so persisting history and persisting model-oriented memory should remain separate concerns in your schema too. ([Home][5])

## 6. Base Entity Design

Create a common base class for audit fields.

```java id="v7j2n0"
@MappedSuperclass
@EntityListeners(AuditingEntityListener.class)
public abstract class BaseAuditEntity {

    @CreatedDate
    @Column(name = "created_at", nullable = false, updatable = false)
    private Instant createdAt;

    @LastModifiedDate
    @Column(name = "updated_at", nullable = false)
    private Instant updatedAt;

    @Column(name = "created_by", length = 100)
    private String createdBy;

    @Column(name = "updated_by", length = 100)
    private String updatedBy;

    @Version
    @Column(name = "row_version", nullable = false)
    private Long rowVersion;
}
```

### Why

* common audit metadata
* optimistic locking support for config records
* consistent DDL generation

Use `@Version` on configuration entities where concurrent edits matter.

## 13. Transaction Design

Spring Data JPA supports transaction configuration, and query methods need explicit configuration if they require transactional semantics. ([Home][3])

### Recommended transaction boundaries

#### Read-only config fetch

```java id="t2q2h4"
@Transactional(readOnly = true)
public RawAssistantConfigBundle load(String assistantCode, String tenantId) { ... }
```

#### Save conversation message + update session timestamp

```java id="o8qm4r"
@Transactional
public void persistAssistantResponse(...) { ... }
```

#### Save execution audit

```java id="6r3qcd"
@Transactional
public void recordExecution(...) { ... }
```

### Important rule

Do not keep one large transaction open across:

* prompt assembly
* LLM invocation
* tool calls
* RAG retrieval

That would create long-lived DB transactions and poor lock behavior.

Instead:

* read config in small read-only transaction
* execute AI call outside DB transaction
* persist results afterward in short write transaction

## 14. JPA Mapping Rules

### Use `LAZY` for relationships by default

For `@ManyToOne` and `@OneToMany`, prefer lazy loading.

Reason:

* avoids unnecessary graph loading
* better performance for config and history queries

### Avoid large bidirectional graphs

Do not create deep cyclical entity graphs unless absolutely necessary.

Good:

* `ChatMessageEntity -> ChatSessionEntity`

Avoid:

* large graph with assistant → policies → history → audits all navigable bidirectionally

### Use projections for read-heavy screens

For admin screens / history lists, use projections instead of loading full entities.

Example:

```java id="mglazc"
public interface ChatSessionSummaryProjection {
    String getSessionId();
    String getAssistantCode();
    String getUserId();
    Instant getLastMessageAt();
    SessionStatus getStatus();
}
```

## 17. Indexing Strategy

Recommended indexes:

### Config tables

* `assistant_code`
* `assistant_id + active`
* `tenant_id + assistant_id + active`

### Conversation tables

* `session_id + created_at`
* `tenant_id + user_id`
* `tenant_id + assistant_code`

### Audit tables

* `request_id`
* `session_id`
* `tenant_id + assistant_code`
* `started_at`

### Why

These are the dominant access patterns:

* load assistant config
* load active policies
* fetch session history in order
* inspect execution/audit by request/session/assistant

## 18. Schema Evolution Guidelines

Since prompts, routes, and policies will evolve, design for versioning.

### Recommended rules

* assistant master keeps `config_version`
* prompts keep explicit `version`
* every execution record stores effective `config_version`
* do not hard delete prompt versions unless necessary
* deactivate rather than overwrite critical config

This preserves reproducibility for audit and debugging.

## 19. Read / Write Patterns

## 19.1 Config reads

High-frequency, mostly cached.

Pattern:

* DB → config layer cache → resolved runtime config

## 19.2 Conversation writes

Moderate-frequency.

Pattern:

* save user message
* save assistant message
* update session last activity

## 19.3 Audit writes

High-value, append-only.

Pattern:

* insert start record
* update completion record
* insert tool/RAG sub-audits

### Recommendation

Audit tables should behave mostly append/update, not mutable business objects.

## 20. Interaction with Spring AI

### Chat memory

Spring AI provides chat memory abstractions and chat-memory advisors; if you choose JDBC-backed memory later, keep that in the memory layer, not mixed into business conversation tables unless intentionally designed. ([Home][5])

### Chat execution

Spring AI’s `ChatClient` supports synchronous and streaming interaction, but your conversation history and audit persistence remain application concerns and should be stored through your JPA layer. ([Home][8])

### Observability

Spring AI can emit observations and optionally log prompt/completion data, but because prompt/completion logs may contain sensitive data and are disabled by default, your persistence layer should store only what you intentionally audit. ([Home][4])
