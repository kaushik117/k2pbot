# Low Level Design — Config Layer

## 1. Purpose

The Config Layer is responsible for loading, validating, caching, resolving, and serving all runtime configuration required by the generic chatbot platform.

This layer must support:

* generic chatbot behavior
* specialization at runtime
* tenant-aware assistant configuration
* configurable prompts
* configurable tools
* configurable RAG knowledge bases
* configurable memory behavior
* configurable model-routing policies

This design aligns with Spring AI’s current architecture, where the actual chatbot execution is assembled around `ChatClient`, Advisors, tool calling, chat memory abstractions, and vector-store-backed retrieval. The config layer should therefore provide clean runtime-ready configuration objects to the orchestration layer rather than mixing config lookup with AI execution. ([Home][1])

---

## 2. Scope of the Config Layer

The Config Layer covers:

* assistant master configuration
* prompt templates and prompt versions
* model routing rules
* memory policy
* RAG policy
* tool policy
* safety / response policy
* knowledge base registration metadata
* config validation
* config caching
* runtime config resolution
* optional config fallback logic

It does **not** execute:

* model calls
* RAG retrieval
* tool calls
* memory persistence logic itself

Those belong to orchestration, RAG, tools, and memory layers respectively.

### Config package responsibility split

* `config.api` → public interfaces exposed to other layers
* `config.model` → domain config models used by runtime
* `config.entity` → persistence entities
* `config.repository` → Spring Data repositories
* `config.loader` → loading raw config from DB/config source
* `config.resolver` → merges configs and runtime overrides
* `config.validator` → validates config integrity
* `config.cache` → caching layer
* `config.mapper` → entity-to-domain and domain-to-DTO mapping
* `config.service` → primary service orchestration for config retrieval
* `config.exception` → config-specific exceptions
* `config.support` → enums, constants, utility classes

## 4. Design Principles

## 4.1 Runtime-ready configuration model

The config layer should return **resolved configuration**, not raw DB rows.

Bad:

* caller loads assistant row
* caller loads prompt row
* caller loads tool rows
* caller manually merges them

Good:

* caller requests one `ResolvedAssistantConfig`
* config layer returns a fully validated runtime object

---

## 4.2 Separation of persistence model and runtime model

Keep:

* DB entities separate from
* runtime configuration objects

Reason:

* persistence structure changes over time
* runtime objects should stay stable and optimized for orchestration
* easier versioning and migration

---

## 4.3 Validation before exposure

No config should be returned to runtime unless:

* structurally valid
* semantically valid
* referentially valid
* active/enabled as required

---

## 4.4 Tenant-aware resolution

The same assistant code may have:

* global default config
* tenant-specific overrides

The config layer must resolve config in this order:

1. platform default
2. assistant default
3. tenant override
4. request runtime override

---

## 4.5 Immutable runtime config

Returned resolved config objects should be immutable.

Reason:

* avoids accidental mutation across requests
* safer for caching
* simpler debugging

---

## 5. Main responsibilities and flow

### Input

* `assistantCode`
* `tenantId`
* optional `channel`
* optional `locale`
* optional runtime override payload

### Output

* `ResolvedAssistantConfig`

### Internal flow

1. load assistant master
2. load active prompt version
3. load model routing policy
4. load memory policy
5. load RAG policy
6. load tool policy
7. load safety/response policy
8. load tenant-specific overrides if present
9. merge config hierarchy
10. validate resolved config
11. cache resolved output
12. return immutable runtime-ready object

## 6.5 Memory config

Spring AI separates `ChatMemory` behavior from `ChatMemoryRepository` storage, and by default auto-configures a `MessageWindowChatMemory` with in-memory storage unless another repository is configured. That makes it sensible for your config layer to express memory as policy, while the memory layer later translates that into Spring AI memory objects. ([Home][2])

```java
public final class ResolvedMemoryConfig {
    private final boolean enabled;
    private final MemoryStoreType storeType;
    private final Integer messageWindowSize;
    private final Integer ttlMinutes;
    private final boolean persistChatHistory;
    private final boolean summarizeOldMessages;
}
```

---

## 6.6 Tool config

Spring AI supports runtime tool registration on `ChatClient`, and tool execution is handled through the request/response loop where the model requests the tool, the application executes it, and the result is returned back to the model. Your config layer should therefore only define which tools are allowed and how they may be used. ([Home][3])

```java
public final class ResolvedToolConfig {
    private final boolean enabled;
    private final List<ResolvedToolDefinition> allowedTools;
    private final boolean allowRuntimeSubsetSelection;
    private final Integer maxToolCallsPerRequest;
    private final Integer toolTimeoutMs;
}
```

```java
public final class ResolvedToolDefinition {
    private final String toolName;
    private final ToolType toolType;
    private final boolean enabled;
    private final boolean requiresApproval;
    private final Integer timeoutMs;
}
```

## 12. Runtime override model

This is request-scoped and should not be persisted as base config.

```java
public class RuntimeConfigOverride {
    private String knowledgeBaseId;
    private MemoryStoreType memoryStoreType;
    private List<String> enabledToolNames;
    private String modelHint;
    private Boolean streamingEnabled;
}
```

### Rule

Runtime override should be treated as a **restricted patch**, not as unrestricted config replacement.

Example:

* allowed to switch KB from approved list
* allowed to choose subset of tools
* not allowed to inject arbitrary new tool names

## 13. Validation design

## 13.1 Validation categories

### Structural validation

Checks:

* required config sections present
* no null critical fields
* values in allowed ranges

### Referential validation

Checks:

* referenced knowledge base exists
* referenced model names belong to allowed provider/model registry
* tools exist in tool registry

### Semantic validation

Checks:

* if grounded mode true, RAG must be enabled
* if citationRequired true, citationsEnabled should normally be true
* messageWindowSize > 0 when memory enabled
* tool timeout must be positive
* model route priorities must be unique or at least deterministic

### Policy validation

Checks:

* runtime overrides cannot violate tenant policy
* blocked tools not present in allowed list
* inactive assistant cannot be returned

## 14. Resolver design

The resolver is the most important internal component in config.

## 14.1 Merge order

```text
Platform defaults
   → Assistant defaults
      → Tenant overrides
         → Runtime overrides
```

## 14.2 Merge strategy by section

### Prompt

* assistant prompt = base
* tenant override may replace or append guardrails
* runtime override normally should not replace prompt template directly

### Model routing

* assistant defines all routes
* tenant may limit model pool
* runtime may only send hint

### RAG

* assistant defines default KB
* tenant may override KB
* runtime may select alternate allowed KB

### Memory

* assistant default may be DB
* tenant may force in-memory or disable
* runtime may request smaller scope if allowed

### Tools

* assistant defines allowlist
* tenant may narrow allowlist
* runtime may choose subset only

## 15. Cache design

## 15.1 What to cache

Cache:

* resolved base config per tenant + assistant

Do not strongly cache:

* highly dynamic runtime overrides

## 15.2 Cache key

```text
tenantId::assistantCode::base
```

Then apply request-time override on top.

## 15.3 Eviction triggers

* assistant updated
* prompt version changed
* KB policy changed
* tool policy changed
* manual admin invalidation

## 15.4 TTL

Suggested:

* 5 to 15 minutes initially
* plus explicit event-driven invalidation

## 19. Example end-to-end flow for config retrieval

```text
1. Orchestration layer calls AssistantConfigProvider.getResolvedConfig()
2. Provider checks cache
3. Cache miss
4. Loader fetches raw entities from DB
5. Raw validator validates entity presence and references
6. Resolver merges assistant + tenant + runtime override
7. Resolved validator validates final runtime policy
8. Provider stores base resolved config in cache
9. Provider returns immutable ResolvedAssistantConfig
```

## 20. Non-functional considerations

### Performance

* DB round-trips should be minimized
* ideally fetch full bundle in one service call path
* caching essential for hot assistants

### Consistency

* config should be versioned
* every response/audit record should store config version used

### Debuggability

* expose “resolved config snapshot” in admin/debug API
* mask sensitive values

### Security

* prevent arbitrary runtime overrides
* tenant isolation mandatory in override resolution

### Observability

Spring AI emits observability data for `ChatClient`, Advisors, and tools, but config resolution itself is your responsibility and should therefore emit custom application metrics such as cache hit/miss, config load time, and validation failure count. Spring AI also cautions that logging prompt content can expose sensitive data, so your config layer should treat prompt templates and resolved prompt snapshots as sensitive artifacts. ([Home][5])

---

## 21. Recommended implementation order

Build in this order:

1. entities + repositories
2. raw bundle loader
3. runtime config models
4. resolver
5. validation rules
6. provider façade
7. cache
8. admin invalidation support
9. config version audit support
