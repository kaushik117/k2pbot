# Low Level Design — Orchestration Layer

## 1. Purpose

The Orchestration Layer is the **runtime control layer** of the chatbot platform.

Its responsibility is to:

* receive a chat request
* load resolved assistant configuration
* build execution context
* choose the execution path
* invoke the selected model through Spring AI
* coordinate memory, RAG, tools, and response shaping
* persist runtime execution metadata
* return synchronous or streaming responses

This layer is the **central runtime engine**. It does not own assistant configuration storage, vector search implementation, tool implementation, or memory persistence implementation. Instead, it coordinates those layers.

This design maps naturally to Spring AI because `ChatClient` is the primary fluent API for chat execution, supports synchronous and streaming interaction, and is designed to be enhanced through Advisors. Spring AI also documents working with multiple chat models, which is important for your configurable model-selection requirement. ([Home][1])

---

## 2. Scope of the Orchestration Layer

The orchestration layer covers:

* chat request intake
* request validation
* config retrieval from config layer
* runtime execution-context creation
* model-routing invocation
* prompt assembly invocation
* advisor chain assembly
* tool enablement for the request
* memory enablement for the request
* RAG enablement for the request
* synchronous or streaming model execution
* execution audit capture
* exception translation
* fallback coordination

It does **not** cover:

* DB config loading
* vector retrieval algorithm implementation
* tool business logic implementation
* memory repository implementation
* API controller serialization details

### Package responsibility split

* `orchestration.api` → interfaces exposed to controller/web layer
* `orchestration.model` → execution models, context objects, request/response wrappers
* `orchestration.service` → façade services used by web/API layer
* `orchestration.executor` → request execution classes
* `orchestration.assembler` → builds advisors, prompt payloads, execution request specs
* `orchestration.advisor` → orchestration-owned custom advisors
* `orchestration.support` → helper classes, policies, constants
* `orchestration.exception` → runtime orchestration exceptions
* `orchestration.mapper` → mapping internal result objects to API response objects

## 4. Design Principles

## 4.1 One orchestration entry point

All chat execution should enter through one orchestration façade.

This avoids:

* duplicated runtime logic
* inconsistent execution policies
* fragmented audit handling

---

## 4.2 Request-scoped execution

Every request should create a fresh execution context.

Do not share mutable execution state across requests.

---

## 4.3 Assemble, do not hardcode

The orchestration layer should assemble execution from:

* resolved config
* runtime override
* execution context
* channel metadata

It should not hardcode assistant-specific branches.

Bad:

```java
if ("collectionsBot".equals(assistantCode)) { ... }
```

Good:

```java
resolvedConfig.getToolConfig()
resolvedConfig.getRagConfig()
resolvedConfig.getModelRoutingConfig()
```

---

## 4.4 ChatClient built per selected model path

Spring AI supports creating `ChatClient` instances programmatically and also documents how to handle multiple chat models when you need different models for different tasks or fallback/A-B scenarios. That fits your runtime model-routing requirement. ([Home][1])

So orchestration should not assume one single global `ChatClient`.

Instead, it should:

* select model first
* then obtain or build the correct `ChatClient`

---

## 4.5 Advisors are the primary extension seam

Spring AI’s Advisors API is explicitly intended for intercepting, modifying, and enhancing AI-driven interactions. That makes advisors the right extension point for memory, RAG, audit, policy enforcement, and response shaping. ([Home][2])

---

## 4.6 Orchestration owns runtime control, not domain logic

The orchestration layer may decide:

* whether tools are enabled
* whether RAG is required
* whether a fallback is needed

But it should not implement:

* loan calculation logic
* policy lookup logic
* retrieval ranking internals

---

## 5. Main Runtime Flow

### Input

* assistant code
* tenant ID
* session ID
* user ID
* user message
* channel/context metadata
* runtime override

### Output

* normalized response payload
* execution metadata
* optional citations
* optional tool execution summary
* optional streaming chunks

### End-to-end flow

1. validate request
2. fetch resolved assistant config
3. construct execution context
4. run model routing
5. assemble prompt
6. assemble advisors
7. resolve tools for request
8. resolve memory strategy
9. resolve RAG strategy
10. build execution spec
11. invoke Spring AI `ChatClient`
12. process response
13. persist audit/history
14. return final response

## 10. ChatClient management

## 10.1 Why registry is needed

Spring AI supports:

* autoconfigured `ChatClient.Builder`
* programmatic client creation
* multiple `ChatClient` instances across models when needed ([Home][1])

Since your system routes requests to different models, orchestration needs a dedicated abstraction.

## 10.2 `ChatClientRegistry`

```java
public interface ChatClientRegistry {
    ChatClient getClient(String provider, String modelName);
}
```

### Implementation idea

```java
@Component
public class DefaultChatClientRegistry implements ChatClientRegistry {

    private final Map<String, ChatClient> clients;

    @Override
    public ChatClient getClient(String provider, String modelName) {
        return clients.get(provider + ":" + modelName);
    }
}
```

### Alternative

Store `ChatModel` and build `ChatClient` on demand:

```java
public interface ChatModelRegistry {
    ChatModel getModel(String provider, String modelName);
}
```

Then:

```java
ChatClient.create(chatModel)
```

### Recommendation

For your design:

* keep a `ChatModelRegistry`
* create `ChatClient` on demand if configuration per request varies heavily
* use prebuilt `ChatClient` only if per-model defaults are stable

## 11. Advisor chain design

Spring AI’s Advisors API exists to intercept and enhance interactions, and RAG support in Spring AI is also advisor-driven. ([Home][2])

## 11.1 `AdvisorChainFactory`

```java
public interface AdvisorChainFactory {
    List<Advisor> createAdvisors(ExecutionContext context);
}
```

## 11.2 `DefaultAdvisorChainFactory`

```java
@Component
public class DefaultAdvisorChainFactory implements AdvisorChainFactory {

    private final MemoryAdvisorFactory memoryAdvisorFactory;
    private final RagAdvisorFactory ragAdvisorFactory;
    private final AuditAdvisorFactory auditAdvisorFactory;
    private final PolicyAdvisorFactory policyAdvisorFactory;

    @Override
    public List<Advisor> createAdvisors(ExecutionContext context) {
        List<Advisor> advisors = new ArrayList<>();

        advisors.addAll(policyAdvisorFactory.create(context));
        advisors.addAll(memoryAdvisorFactory.create(context));
        advisors.addAll(ragAdvisorFactory.create(context));
        advisors.addAll(auditAdvisorFactory.create(context));

        return advisors;
    }
}
```

---

## 11.3 Recommended advisor ordering

```text
1. Policy / validation advisor
2. Prompt enrichment advisor
3. Memory advisor
4. RAG advisor
5. Tool-call advisor / tool execution observation
6. Audit / trace advisor
7. Response post-processing advisor
```

### Why order matters

* policies should run early
* memory should be available before model call
* RAG should augment prompt before model call
* audit should capture enriched execution

## 12. Model selection integration

The orchestration layer should not implement the model-selection logic itself. It should call a dedicated routing service.

```java
public interface ModelRoutingService {
    ModelSelectionResult select(ChatRequest request, ResolvedAssistantConfig config);
}
```

### Flow

1. orchestration loads config
2. orchestration invokes model routing
3. routing returns selected provider/model/temperature/tokens
4. orchestration uses result to obtain correct `ChatClient`

Spring AI’s documentation explicitly describes working with multiple chat models and also includes routing among effective agent workflow patterns, so this separation is aligned with the framework’s intended usage patterns. ([Home][1])

## 13. Prompt assembly integration

Orchestration should also delegate prompt construction.

```java
public interface PromptAssemblyService {

```

### Why

Prompt construction is logically separate from runtime control.

Orchestration should not concatenate prompt strings manually.

## 15. Memory integration

Spring AI’s chat memory abstraction is separate from storage repositories, and the default setup uses `MessageWindowChatMemory` with in-memory repository unless another repository is configured. ([Home][4])

## 15.1 `MemoryAdvisorFactory`

```java
public interface MemoryAdvisorFactory {
    List<Advisor> create(ExecutionContext context);
}
```

### Responsibility

* if memory disabled → return empty list
* if memory enabled → create correct memory advisor chain

### Important orchestration rule

The orchestration layer should ask the memory layer for the correct advisor or memory object, not construct repository implementations itself.

---

## 21.2 Failure policy

### Config resolution failure

* fail fast
* do not call model

### Model selection failure

* fail fast or fallback if assistant policy allows

### Tool policy violation

* fail before model call

### Tool runtime failure

* either fail execution or continue based on assistant tool policy

### RAG unavailable but grounded required

* fail fast

### Memory repository unavailable

* degrade to stateless only if assistant policy allows

## 24. Spring AI integration notes for orchestration

### 24.1 `ChatClient`

Use as the final execution API. It supports:

* fluent prompt construction
* synchronous calls
* streaming calls
* model options at prompt/request level ([Home][1])

### 24.2 Advisors

Use for:

* memory
* RAG
* policy
* audit
* formatting
* request enrichment ([Home][7])

### 24.3 Tool calling

Keep tool resolution outside orchestration business logic, but let orchestration decide the request’s final allowed tool set. ([Home][3])

### 24.4 Observability

Use Spring AI’s built-in metrics/tracing where available, and add application-level execution audit around it. ([Home][6])

### 24.5 MCP

If some tools come from MCP servers, orchestration should still treat them as tools under the same allowlist/policy model. Spring AI positions MCP as a standard way for models to interact with external tools and resources. ([Home][8])

## 26. My recommendation for your project

For your platform, I would keep the orchestration layer **thin but strict**:

* one façade for sync
* one façade for streaming
* one context factory
* one plan factory
* one executor
* one advisor-chain factory
* one audit service

That gives you the cleanest runtime control without turning orchestration into a god-class.

The next section that should logically follow is **Prompt Layer / Model Routing Layer**, but based on your list, the most useful next step is probably **API request/response schemas** or **database schema tables with field definitions**.
